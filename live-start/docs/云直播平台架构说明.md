## 三大模块的边界与职责详解

### 1. Java 后端业务 (API Server for Users)

这是整个平台面向**最终用户（观众和主播）**的“引擎”。

**边界**：
*   **上游**：接收来自**前台界面**的所有 HTTP API 请求和 WebSocket 连接。
*   **下游**：连接并操作**数据库 (MySQL, Redis)**，并调用**阿里云直播的 API**。
*   它绝对不应该包含任何前端代码 (HTML/CSS/JS)，也不应该处理任何与后台管理相关的特定逻辑。

**核心职责**：**处理所有与普通用户和主播直接相关的业务逻辑。**
1.  **用户身份与认证**：
*   处理用户注册、登录、登出。
*   生成和验证 JWT (JSON Web Token)，管理用户会话。
*   提供获取当前登录用户信息的接口 (`/api/users/me`)。
2.  **直播间核心逻辑**：
*   **调用阿里云直播 API**：为认证通过的主播创建直播间，并生成唯一的**推流地址**。
*   为观众提供**播放地址**。
*   接收并处理来自阿里云直播的回调通知（如直播开始、结束、录制完成等）。
3.  **实时互动**：
*   提供一个 **WebSocket 服务器**，用于处理和广播实时弹幕、礼物赠送动画通知、观众进出房间等消息。
4.  **业务功能实现**：
*   处理关注/取关逻辑。
*   处理礼物赠送，更新用户钱包余额和主播收益。
*   提供获取直播列表、分类列表、回放列表等数据的接口。
5.  **安全性**：
*   对所有需要登录才能访问的接口进行权限校验。

*   **设计特点**：
*   **高并发、低延迟**：需要为大量用户同时在线提供服务，接口响应和 WebSocket 通信必须快。
*   **无状态**：每个 API 请求都应通过 JWT 独立认证，便于水平扩展。

 ### 2. Java 后台管理业务 (Admin API Server)

 这是平台**内部运营和管理人员**使用的“指挥中心”。

**边界**：
*   **上游**：接收来自**后台管理界面**的 HTTP API 请求。
*   **下游**：连接并操作**数据库 (MySQL, Redis)**。它拥有比业务后端**更高**的数据库权限（例如，可以直接修改用户状态、删除直播间等）。
*   **它可以是一个完全独立的应用，也可以和业务后端共享部分底层的 Service 和 DAO，但 Controller 和安全策略必须是完全隔离的。**

**核心职责**：**提供数据管理和平台监控的接口，供授权的管理员使用。**
1.  **用户管理**：
*   提供查询所有用户列表（支持分页、按用户名/手机号搜索）的接口。
*   提供封禁/解封用户、修改用户角色（如将普通用户提升为主播）的接口。
2.  **内容管理**：
*   管理直播分类。
*   审查直播间列表，提供强制关闭某个直播间（“超管权限”）的接口。
*   管理直播回放视频（如删除违规回放）。
3.  **财务与运营管理**：
*   管理礼物目录（新增/修改礼物及其价格）。
*   查看礼物交易流水、用户充值记录。
4.  **数据统计与监控**：
*   提供获取平台核心数据（如日活跃用户、总收入、在线人数峰值）的接口，用于生成仪表盘。
5.  **安全性**：
*   实现严格的**基于角色的访问控制 (RBAC)**。只有拥有特定权限的管理员才能调用相应的接口。

**设计特点**：
*   **安全第一**：这是最高优先级。所有接口都必须有严格的权限校验。
*   **数据一致性**：操作通常涉及对数据的直接修改，需要保证事务的完整性。
*   **低并发**：同时使用后台的管理员数量有限，对并发性能要求不高。

### 3. 前台界面 (Frontend UI)
  
这是用户直接看到和交互的“脸面”，运行在用户的浏览器中。
  
**边界**：
*   **它是一个纯粹的客户端应用**。它不包含任何业务逻辑，也不直接连接数据库。
*   它的**唯一数据来源**是通过 HTTP/WebSocket 与 **Java 后端业务 (API Server)** 进行通信。
*   **它绝对不应该，也绝对不能够调用后台管理业务的任何接口。**
  
**核心职责**：**负责渲染界面、响应用户操作，并与业务后端进行数据交换。**
1.  **页面渲染**：
*   展示主页、直播列表页、直播观看页、个人中心等所有面向用户的页面。
2.  **用户交互**：
*   处理用户的点击、滚动、输入等操作。
*   显示登录/注册表单，并将用户输入的数据发送给后端业务 API。
3.  **数据展示**：
*   调用后端业务 API 获取数据（如直播列表），然后将数据渲染成用户可见的列表。
4.  **流媒体播放**：
*   **集成视频播放器 SDK** (如 Aliplayer, DPlayer)。
*   从后端获取播放地址，并将其交给播放器进行播放。
5.  **实时通信**：
*   建立到后端业务 WebSocket 服务器的连接。
*   发送弹幕消息，并接收来自服务器的广播消息（新弹幕、新礼物）并将其显示在界面上。
  
**设计特点**：
*   **用户体验 (UX)**：流畅度、响应速度、界面美观是最高优先级。
*   **状态管理**：需要管理复杂的前端状态（如用户信息、当前直播间状态、弹幕列表等）。
  
---
  
### 总结：为什么这样划分是优雅的？
  
1.  **安全隔离**：最核心的好处。将管理后台与用户业务彻底分开，即使业务服务器出现漏洞，也几乎不会影响到拥有最高权限的管理后台，反之亦然。
2.  **独立开发与部署**：三个团队（前端、业务后端、管理后端）可以并行开发、测试和部署，互不干扰，极大地提高了开发效率。
3.  **技术栈灵活性**：您可以为每个部分选择最适合的技术。例如，业务后端用 Java，管理后端因为开发快可以用 Node.js，前端用 Vue，它们之间通过 API 解耦，完全可行。
4.  **按需伸缩**：用户业务（API Server）面临高并发，可以部署 100 个实例；管理后台（Admin Server）并发低，可能只需要部署 2 个实例。这种独立的伸缩能力可以极大地节约服务器成本。
